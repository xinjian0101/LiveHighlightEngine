# Support

## Start here

Before opening an issue, review:

- [README](README.md) for commands and current capabilities
- [About](ABOUT.md) for scope and project boundaries
- [Architecture](docs/ARCHITECTURE.md) for processing stages
- [Scoring Guide](docs/SCORING_GUIDE.md) for profile calibration
- [Error Handling](docs/ERROR_HANDLING.md) for validation behavior

## Bug reports

Use the structured **Bug report** form for reproducible defects in event loading, configuration, scoring, merging, ranking, or exports.

A useful report contains:

- exact commit or release;
- operating system and Python version;
- minimal synthetic JSONL input;
- scoring configuration;
- complete reproduction command;
- expected and actual result;
- error output without private paths or credentials.

## Feature requests

Use the **Feature request** form. Describe the workflow problem, proposed event contract, acceptance criteria, benchmark impact, and backward-compatibility considerations.

## Questions that belong in issues

Open an issue when the question reveals unclear behavior, missing documentation, a reproducible compatibility problem, or a proposal that could improve the project for multiple users.

## Questions outside project scope

This repository cannot provide:

- review of private source recordings;
- copyright or privacy clearance;
- automatic publishing decisions;
- guarantees that a scoring profile is optimal for every content category;
- support for uncommitted local forks without a reproducible diff.

## Security and privacy

Do not upload source video, credentials, private transcripts, personal data, or internal directory structures. Replace all examples with synthetic values.

## Response expectations

Support is community-maintained and best effort. Clear reproduction steps, small fixtures, and focused requests are easier to review than broad requests without examples.
